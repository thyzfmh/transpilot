# AI 辅助源码迁移的反幻觉机制

## 背景

这篇文档来自太白在 AI 辅助研发过程中的经验总结。

在实践中，AI 很擅长快速阅读代码、生成实现、补测试和整理文档，但它也会在证据不足时给出过度自信的判断。尤其在源码迁移、复杂重构、跨语言改写这类任务里，AI 的错误不一定表现为编译失败，更多时候会表现为“看起来能跑，但行为已经偏了”。

因此，太白把反幻觉经验沉淀为一套通用机制：不是要求 AI 永远不犯错，而是通过源码证据、独立 Oracle、差分测试、原子任务、充分测试和独立评审，把错误尽早暴露出来，并在验证策略变弱时让用户明确知情。

## 1. 为什么源码迁移特别怕幻觉

AI 辅助源码迁移里，最危险的幻觉不是“代码编译不过”。编译失败通常很快就能发现。真正危险的是：

- 代码能编译；
- 单元测试也通过；
- API 看起来相似；
- 但真实行为已经偏离源项目。

这种问题在大项目迁移中尤其致命。迁移通常会持续很多轮，一个早期行为误判如果没有被发现，后续模块会继续建立在这个错误假设上，最后形成系统性偏差。

反幻觉机制不是简单提醒 AI “不要胡说”，而是把 AI 的关键判断放进一套可追踪、可验证、可回滚的工程流程里。

核心原则是：

```text
AI 不能凭印象翻译。
AI 不能自己写实现、自己编测试预期、自己宣布通过。
每个关键行为都必须有证据、独立 Oracle、测试和评审。
```

这里的目标不是让 AI 永远不犯错，而是让错误尽早暴露，让验证不能被 AI 自己污染，让用户在风险升高时明确知情。

## 2. 八层防线

一套实用的反幻觉机制，至少应该包含八层防线：

1. 源码证据约束理解；
2. 幻觉类型识别与对治；
3. 独立 Oracle 约束测试；
4. 差分测试约束行为等价；
5. 验收计划约束项目可信度边界；
6. 原子任务计划约束执行漂移；
7. 充分测试约束边界行为；
8. 独立评审约束完成声明。

可以把它理解成一条流水线：

```text
理解源码
  -> 拆分原子任务
  -> 写任务计划
  -> 建立独立 Oracle
  -> 翻译实现
  -> 差分/回归/边界测试
  -> 独立评审
  -> 用户只在风险升级时介入
```

自动化应该减少用户操作负担，但不能替用户隐藏风险判断。普通用户不需要理解所有术语，但系统必须在关键风险点给出清楚的人话解释：

```text
检测到源项目无法运行。
行为验证将从“运行源项目对照”降级为“静态源码对照”。
这会降低最终可信度上限。
是否继续？
```

这是产品质量红线：**减少操作，不减少知情权**。

## 3. 第一层：源码证据约束理解

AI 在迁移非平凡逻辑前，必须先回答六个问题：

1. 我描述的源码行为，有没有源码片段、行号或调用图证据？
2. 我使用的目标语言 API，有没有官方文档、仓库用法或最小编译证明？
3. 所有数字、字符串、错误码、默认值，有没有源码来源？
4. 调用关系有没有做源/目标双向核对？
5. 能不能提出一个源项目和迁移项目可能行为不一致的反例？
6. 测试 expected 值来自哪里？是否独立于实现 AI？

这套问题的目标，是把 AI 从“我觉得源码是这样”拉回到“我看到了源码证据”。

典型要求：

```text
错误：这个 map 是线程安全的，所以用普通 HashMap 加锁就行。
正确：源代码使用原子 LoadOrStore 语义，目标实现必须保留并发重复插入时只有一个 winner 的行为。
证据：源码调用点、并发路径、重复插入测试。
```

证据不只包括源码正文，也包括：

- 官方测试；
- 文档或注释；
- 调用图；
- 编译条件；
- fixture；
- 历史 bug 和回归测试；
- 源项目真实运行输出。

没有证据的行为描述只能作为假设，不能作为验收依据。

## 4. 第二层：识别五类常见幻觉

源码迁移中的幻觉通常不是随机发生的，而是集中在几类高风险位置。

| 幻觉类型 | 表现 | 对治方式 |
|---|---|---|
| API 编造 | 使用不存在或语义不对的目标语言 API | 查官方文档、仓库用法、最小编译验证 |
| 行为臆测 | 按语言印象翻译，而不是按源码行为翻译 | 源码证据 + 行为对照表 |
| 常量伪造 | 端口、超时、错误码、重试次数写错 | 常量必须引用源码来源 |
| 调用关系反转 | 把 A 调 B 翻成 B 调 A，或同步/异步关系错位 | 调用图双向核对 |
| 浅测试误判 | happy path 通过，但边界行为错 | 反例测试、性质测试、差分测试 |

最重要的一点是：**测试通过不等于语义等价**。如果测试只覆盖 happy path，它很可能只是证明 AI 写的错误实现和 AI 写的错误预期相互匹配。

在系统软件、基础库、协议库、C/C++ 项目迁移里，幻觉还会更隐蔽。AI 很容易犯下面这些错：

- 把错误码语义简化成普通异常或 `Result`，但丢失 `errno`、重试、非阻塞返回值；
- 把宏、编译开关、平台差异当成普通常量处理；
- 忽略资源生命周期、后台线程关闭协议、析构顺序；
- 把 C/C++ ownership 约定迁移到安全语言时，错误地提前释放或重复释放；
- 只测单线程 happy path，没有覆盖 timeout、buffer limit、poll、interrupt、backpressure 等行为。

因此不能只问“目标语言代码是否优雅”，还必须问：

```text
这个行为在源项目里到底在哪里定义？
这个行为是否受平台、编译选项、环境变量或时序影响？
这个行为有没有源项目测试、文档或真实运行输出来证明？
目标实现是否保留了同样的失败模式和边界语义？
```

## 5. 第三层：Oracle 独立性

Oracle 是“测试预期值从哪里来”。

反幻觉的铁律是：

```text
AI 写实现 + AI 写 expected = 自己批改作业 = 测试无效。
```

禁止把下面这种测试当成语义等价证明：

```rust
assert_eq!(foo("hello"), "HELLO");
assert_eq!(bar(3), 6);
assert!(result.contains("expected"));
```

这些 expected 值如果不是来自源项目、用户确认、官方文档或独立验证，就只是 AI 的猜测。

合法 Oracle 包括：

- 源项目真实运行结果；
- 源项目官方测试、fixture、record-replay trace；
- 源码静态特征、常量、调用图；
- 用户明确确认的常量或行为；
- 独立测试 agent 在隔离上下文中推导的测试；
- property / metamorphic test，例如幂等性、往返一致性、顺序保持。

Oracle 的可信度可以分级：

| 级别 | Oracle 来源 | 可信度 | 适用场景 |
|---|---|---:|---|
| A | 源项目真实运行输出 | 最高 | 源项目可编译、可运行、可构造输入 |
| B | 源项目官方测试 / fixture / trace | 高 | 复杂 I/O、协议、状态机 |
| C | 源码静态证据 + 常量 + 调用图 | 中 | 源项目暂时跑不起来，但源码结构清晰 |
| D | 独立测试 agent 推导 | 中低 | 只能作为补充，不能单独决定交付 |
| E | 实现 agent 自己写 expected | 无效 | 必须禁止 |

这张表的意义不是给 AI 打分，而是决定交付口径。A/B 级 Oracle 可以支持较高可信度；C/D 级只能说明“当前证据下未发现偏差”；E 级不能进入验收。

## 6. 第四层：差分测试

当源项目可以运行时，最可靠的验证方式是差分测试：

```text
同一个输入 -> 源项目运行 -> 得到 src_out
同一个输入 -> 迁移项目运行 -> 得到 dst_out
断言 src_out == dst_out
```

AI 只允许生成输入，不允许编 expected。expected 必须来自源项目。

示意：

```rust
#[test]
fn diff_static_cases() {
    for input in ["", "a", "hello"] {
        assert_eq!(src_run(input), dst_run(input));
    }
}
```

如果源项目涉及 I/O，则使用 record-replay：先录制源项目真实交互，再在迁移项目里回放。

如果源项目跑不起来，只能降级为静态源码对照、官方测试对照或独立测试 agent。这种降级不能静默发生，必须告知用户，并降低可信度上限。

差分测试不只适合纯函数。对大项目，可以分成几种粒度：

| 粒度 | 验证对象 | 示例 |
|---|---|---|
| 函数级 | 输入输出明确的工具函数 | parse、encode、normalize |
| 模块级 | 有内部状态但边界清楚 | option、message、routing table |
| 进程级 | CLI、daemon、server | 同一命令行参数下比较 stdout/stderr/exit code |
| 协议级 | 网络或 IPC 行为 | record-replay、fixture、packet trace |
| 性质级 | 很难逐项枚举的行为 | 幂等性、顺序保持、资源释放后不可用 |

越靠近系统边界，越应该优先使用源项目真实行为或录制 trace，而不是让 AI 总结 expected。

## 7. 第五层：验收计划防止验证策略静默变化

项目开始前，应该有一个用户确认过的验收计划。它不一定叫 `acceptance-plan.yaml`，但必须回答四个问题：

1. 用户到底要迁移哪些范围？
2. 哪些行为必须等价？
3. 用什么证据证明等价？
4. 什么情况必须暂停并请求确认？

验收计划至少应该包含：

- 迁移范围；
- 不迁移范围；
- Oracle 策略；
- 源项目是否可运行；
- E2E 命令；
- must-pass 用例；
- 允许的可信度下限；
- 禁止的验证方式，例如 AI-derived expected；
- 风险升级条件。

这样可以防止一种很隐蔽的风险：

```text
项目一开始说要运行源项目做行为对照，
后来静默降级成静态源码对照，
最终却仍然按高可信度交付。
```

验收计划的核心不是“多写一个配置文件”，而是固定住项目承诺。没有这个边界，自动化越强，风险越大。因为 agent 很可能会为了继续推进而悄悄缩小范围、替换验证方式、跳过难测模块。

## 8. 第六层：原子任务计划防止执行漂移

大项目不能让 AI 一口气翻译。必须拆成多个原子任务，每个任务有自己的计划文件或任务卡。

一个合格的原子任务计划必须包含：

- 任务目标；
- 用户确认的 scope；
- source evidence；
- functional requirements；
- 精确输入、输出、边界和错误行为；
- atomic tasks；
- acceptance criteria；
- test matrix；
- verification commands；
- review gate。

这一步的价值是：AI 不能边写边改目标，也不能做完后才补测试。它必须先说明：

```text
我要迁移什么；
为什么迁移这些；
每个行为的输入输出是什么；
边界条件是什么；
怎么测试；
怎么判定完成。
```

原子任务应该足够小。判断标准不是“能不能塞进一个 prompt”，而是：

- 有清晰的源码边界；
- 有明确的用户价值或模块价值；
- 有可列举的行为需求；
- 有可执行的验收命令；
- 独立评审能在有限上下文内判断是否完成。

如果一个任务同时包含协议解析、线程模型、内存管理和平台适配，它就太大了。应该拆成多个任务，例如：

```text
task-001: 消息帧数据结构和基本 ownership
task-002: option 读写语义
task-003: send/recv 非阻塞与错误码映射
task-004: poll/timeout/interrupt 行为
task-005: context shutdown 与资源释放
```

每个任务计划既是执行计划，也是最小原子任务说明书。

## 9. 第七层：测试必须充分，禁止敷衍

反幻觉机制必须明确禁止 shallow tests。

每个任务至少要考虑：

- happy path；
- 空输入 / 零值 / null 等价输入；
- 非法输入 / 错误路径；
- 边界大小；
- 重复调用；
- 资源生命周期；
- 并发或顺序敏感行为；
- 源项目已有回归测试。

对于复杂模块，仅有下面这种测试是不够的：

```rust
assert!(result.is_ok());
```

它只能说明“没有报错”，不能证明行为等价。

合格测试应该能回答：

```text
输入是什么？
源项目输出是什么？
迁移项目输出是什么？
边界情况下是否一致？
expected 是否独立于 AI？
```

测试充分性的最低要求可以写成下面这张表：

| 行为类型 | 至少要测什么 |
|---|---|
| 解析/编码 | 空输入、非法输入、最大长度、往返一致性 |
| 状态机 | 初始状态、合法转移、非法转移、重复调用 |
| 资源生命周期 | 创建、使用、关闭、关闭后再用、重复关闭 |
| 错误处理 | 错误码、错误消息、恢复能力、是否保留源项目语义 |
| 并发行为 | 顺序保证、竞态敏感路径、重复执行稳定性 |
| 平台差异 | Linux/macOS/Windows 条件编译、路径、socket、文件描述符 |
| 性能相关行为 | 超时、缓冲区上限、backpressure、批量输入 |

这不是要求每个任务都写海量测试，而是要求测试矩阵与行为风险匹配。风险越高、边界越多、历史 bug 越多，测试就必须越密。

## 10. 第八层：独立评审门禁

实现 agent 不能自己宣布完成。任务完成后，必须交给独立评审检查：

1. 每个 functional requirement 是否有测试；
2. 测试是否覆盖边界和错误场景；
3. Oracle 是否独立；
4. 行为是否有源码证据；
5. 验证命令是否真实运行并通过；
6. 是否存在未解释的行为差异；
7. 是否存在占位符、TODO 或未完成实现。

如果评审认为不达标，执行 agent 必须带着反馈继续当前任务，直到评审门禁通过。

独立评审的职责不是重新实现一遍，而是做质量闸门。它应该优先找这些问题：

- 计划里写了需求，但代码没有实现；
- 代码实现了行为，但测试没有覆盖；
- 测试存在，但 expected 没有独立来源；
- 验证命令没有真实运行；
- 失败被解释成“已知问题”，但没有用户确认；
- 行为差异被文档化了，却没有进入验收计划；
- 任务范围被扩大或缩小，但没有更新计划。

评审结论应该是三态，而不是一句模糊评价：

```text
PASS: 可以进入下一任务。
REVISE: 当前任务继续执行，必须修复反馈。
ESCALATE: 涉及范围、可信度或产品风险，必须让用户确认。
```

这样系统才能一轮一轮往下做，而不是每次都让用户判断技术细节。

## 11. 面向普通用户的交互原则

反幻觉机制不能只服务专家。普通用户不应该被迫理解所有内部指标，但必须知道关键风险。

用户默认看到的状态应该是：

```text
任务 3 / 12
当前目标：option 读写语义
进度：实现 4/6，测试 18/24
风险：中
需要确认：源项目在 macOS 下某个 option 行为与 Linux 不一致
```

而不是：

```text
oracle_mode=static-codegraph
hallucination_score=0.18
parity_delta=-0.11
```

专家信息可以保留在高级模式，但默认状态页必须用普通用户能理解的语言。

好的交互有三个标准：

- 告诉用户发生了什么；
- 告诉用户会影响什么；
- 给出少量清晰选择。

不要把内部指标直接丢给用户，也不要静默替用户做高风险决定。

## 12. 可信度指标

系统不应该只输出“完成”或“失败”，还应该输出可信度构成。建议最少记录这些指标：

| 指标 | 含义 | 风险信号 |
|---|---|---|
| source evidence coverage | 关键行为是否有源码证据 | 低覆盖说明 AI 可能在猜 |
| oracle independence | expected 是否独立于实现 AI | 不独立则测试无效 |
| parity pass rate | 差分测试通过率 | 下跌说明行为偏离 |
| must-pass pass rate | 用户指定关键用例通过率 | 失败必须阻断 |
| task scope drift | 任务是否偏离计划 | 漂移说明执行失控 |
| review revise count | 同一任务被打回次数 | 多次打回需要升级 |
| unresolved behavior gaps | 未解释行为差异数量 | 不能带着差异交付 |

这些指标不是为了让用户看复杂面板，而是为了让系统做自动决策：

```text
如果 oracle independence 失败 -> 阻断。
如果 must-pass 失败 -> 阻断。
如果 parity 明显退步 -> 暂停当前任务。
如果 scope drift 发生 -> 要求更新计划或回退范围。
如果同一任务多次 REVISE -> 升级给用户。
```

## 13. 升级机制

以下情况不能自动吞掉，必须升级给用户：

- Oracle 缺失；
- Oracle 从源项目运行结果降级；
- 出现 AI-derived expected；
- 幻觉风险评分超阈值；
- 多个任务连续出现无源码证据断言；
- parity 反复低于阈值；
- E2E 或 must-pass case 失败；
- 设计歧义影响公开 API；
- 源项目自身行为疑似有 bug。

这体现一个基本产品原则：

```text
自动化是为了减少操作负担，
不是为了替用户隐藏风险判断。
```

升级提示必须短、具体、可选择：

```text
当前任务无法达到高可信度验收。

原因：
源项目无法运行，因此不能做真实行为对照。
目前只能根据源码结构和常量做静态验证。

影响：
这部分迁移可以继续推进，但不能按“行为完全等价”交付。

可选操作：
1. 继续，但标记为中可信度。
2. 先修复源项目构建，让系统重新跑行为对照。
3. 缩小本任务范围，只迁移已有独立测试覆盖的部分。
```

这类交互可以多一点，尤其在项目早期。用户不需要学会所有机制，但需要知道自己每一次“继续”意味着什么。

## 14. 对大 C/C++ 项目的特别要求

大 C/C++ 项目比一般业务代码更容易触发幻觉，因为很多行为不在函数签名里，而藏在编译系统、宏、平台 API、全局状态和 undefined behavior 边缘里。

面对这类项目，至少要额外收集：

- 构建入口：`configure`、`cmake`、`make`、feature flags；
- 平台条件：`#ifdef`、系统调用、socket/file descriptor 差异；
- 公开 ABI：头文件、导出符号、结构体布局、错误码；
- 生命周期：init/shutdown、线程启动停止、资源释放顺序；
- 源项目测试：unit、integration、fuzz、stress、flaky 说明；
- 协议或 wire format：二进制布局、大小端、版本兼容；
- 性能边界：timeout、buffer、backpressure、high-water mark；
- 历史 bug：CHANGELOG、issue、回归测试。

这决定了任务拆分不能只按文件数，也要按行为风险拆。一个文件很小，但如果它定义 ABI 或错误码，就必须单独成为高优先级任务。一个文件很大，但只是平台适配，也可以拆成多个平台任务。

对复杂系统项目，合理的前几步不是马上整仓迁移，而是：

1. 跑通源项目构建和测试；
2. 提取公开 API、错误码、核心数据结构的行为证据；
3. 先选一个窄 scope；
4. 建立源项目 fixture 或 record-replay；
5. 生成原子任务计划；
6. 翻译最小可验证实现；
7. 用差分测试和独立评审卡住验收。

这样做看起来慢，但能避免最危险的情况：AI 快速写出一套“像原项目”的新实现，但真实语义已经偏离源项目。

## 15. 机制总结

通用反幻觉机制可以浓缩成一句话：

```text
用源码证据约束理解，
用独立 Oracle 约束测试，
用原子任务计划约束执行，
用独立评审约束完成声明，
用验收计划约束项目可信度边界。
```

它解决的不是“让 AI 永远不犯错”，而是：

- AI 犯错时能被测试抓住；
- 测试不能由 AI 自己伪造；
- 验证策略变弱时用户必须知道；
- 每个任务的目标和验收标准都可审；
- 最终交付有证据链，而不是只有 AI 的自信。

这才是大规模源码迁移里真正可用的反幻觉机制。
