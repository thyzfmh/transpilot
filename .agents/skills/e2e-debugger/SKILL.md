# E2E 诊断器 (E2E Debugger)

## 概述

当端到端测试失败时，自动诊断失败原因，缩小故障范围到具体模块/函数，
并提供修复建议。从 Taibai 教训 L003（Late E2E）中提炼。

## 核心原则

### 原则 1: E2E 尽早触发
> "第一个功能模块翻译完成后立即建立 E2E" — L003

不要等所有模块都翻译完才运行 E2E。
最小可运行路径确定后就开始。

### 原则 2: 二分法定位
大型项目 E2E 失败时，不要逐行排查。
用二分法快速缩小范围。

### 原则 3: 差异驱动
关注"源项目能通过但 Rust 项目不能通过"的情况。
这意味着翻译引入了行为差异。

## 诊断流程

### Step 1: 失败分类
```
E2E 失败
├── 编译失败 → 类型错误（回到翻译阶段）
├── 运行时 panic → 定位 panic 源（通常是 unwrap/todo!）
├── 输出不匹配 → 行为差异诊断
├── 超时 → 死锁/性能诊断
└── 资源错误 → 环境/配置问题
```

### Step 2: 差异定位（二分法）

```
给定: 失败的 E2E 场景 S
1. 确定 S 经过的模块列表 [M1, M2, ..., Mn]
2. 在中间模块 M(n/2) 插入检查点
3. 对比检查点处的状态:
   - 源项目状态 vs Rust 项目状态
   - 如果相同 → 问题在后半段
   - 如果不同 → 问题在前半段
4. 递归缩小范围，直到定位到单个模块
5. 在模块内继续二分，定位到具体函数
```

### Step 3: 根因分析

**常见根因分类:**

| 分类 | 表现 | 典型原因 |
|------|------|---------|
| 类型差异 | 数值截断/溢出 | 整数类型映射错误 |
| 零值差异 | nil vs None 行为不同 | Option 处理遗漏 |
| 序列化差异 | JSON 字段缺失/多余 | serde 属性配置错误 |
| 并发差异 | 竞态/死锁 | 锁粒度不同 |
| 错误处理差异 | 应返回错误却 panic | unwrap 或 todo! |
| 排序差异 | 集合顺序不同 | HashMap 迭代顺序不确定 |

### Step 4: 修复建议生成

```
对每个定位到的问题，生成:
1. 问题描述（什么行为不一致）
2. 根因（为什么不一致）
3. 修复方案（如何修复）
4. 验证方法（如何确认修复成功）
5. 预防措施（如何避免同类问题再次出现）
```

## 诊断工具箱

### 工具 1: 状态快照对比
```bash
# 在关键点捕获两侧状态
SOURCE_STATE=$(run_source_with_checkpoint $CHECKPOINT)
RUST_STATE=$(run_rust_with_checkpoint $CHECKPOINT)
diff <(echo "$SOURCE_STATE") <(echo "$RUST_STATE")
```

### 工具 2: 序列化输出对比
```bash
# 对比 JSON/YAML 输出（忽略顺序）
jq -S . source_output.json > /tmp/source_sorted.json
jq -S . rust_output.json > /tmp/rust_sorted.json
diff /tmp/source_sorted.json /tmp/rust_sorted.json
```

### 工具 3: 日志级别对比
```
在两侧启用详细日志，对比关键事件序列:
- 函数调用顺序
- 分支选择
- 错误处理路径
```

### 工具 4: 模糊测试
```bash
# 用相同输入驱动两侧，对比输出
cargo fuzz run e2e_compare -- -max_len=4096
```

## 失败模式库

### FM-001: Map 迭代顺序
**表现:** 输出顺序不一致
**根因:** Go map / C hashmap 迭代顺序不确定，但测试依赖顺序
**修复:** 输出前排序，或使用 BTreeMap

### FM-002: 浮点精度
**表现:** 数值微小差异导致断言失败
**根因:** 浮点运算顺序不同导致累积误差
**修复:** 使用 approx_eq 比较，或固定运算顺序

### FM-003: 时间相关测试
**表现:** 超时或时序断言失败
**根因:** 不同语言的定时器精度/调度不同
**修复:** 放宽时间断言，或使用 mock 时间

### FM-004: 默认值差异
**表现:** 字段值为 0/null/empty 时行为不同
**根因:** Go 零值语义 vs Rust Option::None 语义
**修复:** 确保 Default impl 匹配 Go 零值

### FM-005: 错误信息差异
**表现:** 错误消息文本不同
**根因:** 错误格式化方式不同
**修复:** 如果只是消息不同但语义相同，放宽断言

### FM-006: 并发竞态
**表现:** 间歇性失败
**根因:** 翻译改变了并发粒度
**修复:** 分析原始同步逻辑，确保 Rust 等价

## E2E 最小化原则

不要构建巨大的 E2E 测试套件。遵循:

1. **最小路径覆盖:** 选择能覆盖最多模块的最短路径
2. **关键场景优先:** 优先覆盖核心业务流程
3. **错误路径必覆盖:** Happy path + 至少 1 个 error path
4. **渐进扩展:** 随翻译进度扩展 E2E 范围

```markdown
## E2E 诊断报告 — [场景名]

**失败时间:** YYYY-MM-DD HH:MM
**失败类型:** [编译/Panic/输出不匹配/超时]

### 症状
[描述观察到的失败现象]

### 定位过程
1. 经过模块: [M1 → M2 → M3 → M4]
2. 检查点 M2: 状态一致 ✓
3. 检查点 M3: 状态不一致 ✗
4. 定位到: M3::function_name

### 根因
[分类] — [详细描述]

### 修复
```diff
- 旧代码
+ 新代码
```

### 验证
- [ ] 单元测试通过
- [ ] 当前 E2E 场景通过
- [ ] 回归测试通过

### 预防
[记录到反模式库 / 更新翻译技能]
```
